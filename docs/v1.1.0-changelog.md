## 1. 统一工具协议 + 权限前置

### 如何实现

在 `tools_registry.py` 中新增 `ToolMeta` 数据类，为每个工具注册元数据：

```python
@dataclass
class ToolMeta:
    safety_level: SafetyLevel      # READ_ONLY / WRITE / DESTRUCTIVE / DANGEROUS
    is_read_only: bool
    is_concurrency_safe: bool
    timeout_sec: int
    tags: tuple                    # server / log / network / k8s / docker / security / ...
```

提供核心能力：
- `register_tool_meta()` — 工具定义后注册元数据
- `filter_tools_by_safety()` — 按安全等级过滤（模型看到前就剔除危险工具）
- `filter_tools_by_tags()` — 按领域标签过滤（如只暴露 `network` 类工具）
- `get_tool_name()` / `get_tool_meta()` — 统一获取工具名称和元数据

23 个内置工具 + 5 个自由工具全部注册了 metadata，`AgentLoop` 新增 `permission_mode` 参数控制过滤策略。

### 亮点

- **安全前置**：不再等 LLM 调了工具再拦截，而是在工具暴露阶段就按权限过滤，模型看到的工具集就是安全的
- **标签过滤**：支持按领域筛选工具（`server`/`log`/`network`/`k8s`/`docker`/`security`/`runbook`/`general`），可按场景动态暴露能力
- **工具描述增强**：`/tools` 命令现在标注 [只读] / [写操作] 标签，用户一眼识别
- **16 个只读工具 + 7 个写操作工具**，分类清晰

## 2. 上下文注入独立化

### 如何实现

新建 `context_injector.py`，创建 `ContextInjector` 类：

```python
class ContextInjector:
    def collect(user_input: str) -> InjectedContext:
        # 1. 主机上下文：hostname / OS / uptime / 最近巡检数据
        # 2. 任务上下文：最近操作的主机 / 使用的工具
        # 3. 记忆摘要：跨会话的操作回顾
```

- `_collect_host_context()` — 并行收集主机信息（socket.gethostname + uname + uptime + 巡检历史）
- `_collect_task_context()` — 从巡检历史提取最近操作的主机
- `_collect_memory_summary()` — 根据用户输入匹配相关记忆
- 5 分钟缓存，避免每次收集重复开销

集成到 `AgentLoop`：`_build_system_prompt_with_context()` 在创建 Agent 时将上下文追加到 system prompt。

### 亮点

- **LLM 开口前就知道环境状态**：启动时自动注入"当前主机名、OS 版本、运行时间、上次巡检 CPU/MEM/DISK 数据"
- **从拼接到注入**：不再往用户输入里硬塞记忆摘要，而是作为系统级上下文注入到 system prompt
- **缓存机制**：5 分钟内复用已收集的上下文，减少系统调用开销
- **零侵入**：对现有流程无影响，只增加信息量

## 3. 命令/工具系统分离

### 如何实现

新建 `commands.py`，将斜杠命令从 `HybridAgent._handle_slash_command()` 抽到独立模块：

```python
class CommandRegistry:
    def register(name, handler, aliases)  # 注册命令
    def dispatch(cmd)                       # 查找并执行
```

内置命令统一注册：
- `/clear` — 清空历史
- `/history` — 执行摘要
- `/tools` — 工具列表
- `/mode` — 运行模式
- `/status` — **新增**，Agent 状态快照
- `/plugins` — 插件信息
- `/memory` — 记忆查询（支持参数解析）

`HybridAgent` 改为调用 `commands.py` 中的函数，不再硬编码 if-else。

### 亮点

- **职责清晰**：命令系统（用户显式控制）与工具系统（模型隐式调用）分离
- **可扩展**：新增命令只需在 `commands.py` 注册，不用改 `hybrid.py`
- **新增 `/status`**：实时查看 Agent 运行状态（主机、工具模式、引擎、警告数）
- **参数解析统一**：`/memory` 的 `--host`/`--cat`/`--search`/`--date` 参数解析也移入命令模块

## 4. 排查计划动态生成

### 如何实现

在 `planner.py` 中新增 `generate_dynamic_plan()`：

```python
def generate_dynamic_plan(user_input: str) -> ExecutionPlan:
    # 1. 模板匹配优先（6 个预定义模板）
    # 2. 模糊匹配兜底（cpu...高 / 负载...高 正则匹配）
    # 3. 关键词分析动态生成（nginx → 检查服务状态，日志 → 查 err）
    # 4. 未识别方向给通用 plan
```

动态生成规则：
- 服务器资源 → `inspect_server` → `get_top_processes`
- 服务相关 → + 进程状态检查
- 日志相关 → + `query_system_logs priority=err`
- 网络相关 → `ping_host` → `check_port`
- 安全相关 → `scan_ports` → 登录日志
- K8s/Docker → 对应巡检工具

### 亮点

- **从 6 个固定模板 → 无限场景覆盖**：之前"帮我看看服务器好像有问题"这种不在模板里的输入不会展示排查路线，现在会根据关键词自动组装
- **模糊匹配修复**："cpu为什么高"、"负载有点高"等中间插入词语的输入现在能匹配上模板
- **三层兜底**：精确模板 → 动态生成 → 通用 inspect_server，总有一条命中
- **用户确认后可执行**：计划仍保留 `[排查路线: ...]` 格式注入用户输入，LLM 会按此路线推进

## 5. AppState 状态总线

### 如何实现

新建 `state.py`，创建 `AgentStateStore`：

```python
class AgentStateStore:
    session: SessionState  # 集中管理所有状态
    def get/set(key, value)
    def register_hook(key, callback)  # 状态变更监听
    def snapshot() / format_status()
```

管理的状态：`is_running` / `current_host` / `tool_mode` / `permission_mode` / `active_loop_mode` / `last_tool_calls` / `warnings`

`HybridAgent` 初始化 `state_store` 并在每次 Agent 执行后同步状态。

### 亮点

- **从散落字段到统一总线**：之前状态散落在 `self.state` / `self._agent_loop` / `self._stream_callback`，现在集中管理
- **状态变更钩子**：可扩展监控，如监听 `current_host` 变化自动记录审计日志
- **警告队列**：工具执行中的警告（如"连续调用同一工具 3 次"）可积累后统一展示
- **调试友好**：`format_status()` 输出可读状态报告，`snapshot()` 返回 dict 供审计使用

## 6. 工具输出压缩

### 如何实现

新建 `compressor.py`，四级压缩管线：

| 级别 | 策略 | 触发条件 | 说明 |
|------|------|---------|------|
| 1 | 结构化摘要 | 日志类 + >1500 字符 | 保留 error/warning/crit 行，丢弃噪音 |
| 2 | 折叠 | >4500 字符 | 保留首尾，中间用占位符 |
| 3 | 裁剪 | >3000 字符 | 直接截断 |
| 4 | 历史压缩 | >500 字符 | 历史存储用，保留首 2 行 + 统计 |

```python
class OutputCompressor:
    def compress(tool_name, content, max_len) -> CompressedResult
    def compress_for_history(content, max_len=500) -> str
```

集成到 `AgentLoop._run_manual()`：工具执行后先压缩再注入下一轮，超长输出自动选择最优策略。

### 亮点

- **日志智能摘要**：100 行的日志只保留 error/warning 行 + 首尾，信息量不变但字符数减少 80%+
- **折叠不丢信息**：长内容不是简单截断，而是"保留首尾 + 中间折叠 + 统计信息"，LLM 知道"中间有 N 行被折叠了"
- **压缩策略透明**：压缩发生时通过 warning 回调通知用户（"输出已压缩 summarize: 3345 → 484 字符"）
- **历史存储优化**：对话历史不再硬截断 500 字符，而是按行裁剪 + 保留统计信息

## 7. 结构化提问

### 如何实现

新建 `ask_user.py`，创建 `AskUserParser`：

```python
class AskUserParser:
    def parse(tool_name, content) -> AskUserResult
    def format_for_display(result) -> str
```

识别规则覆盖三类场景：
- **SSH 连接失败**：提取用户名/密钥/端口等引导项，展示为选项列表
- **K8s 连接失败**：展示 kubeconfig 路径选项（默认/K3s/自定义）
- **依赖未安装**：展示"是，帮我安装 / 否，我自己安装"选项

集成到 `HybridAgent.process()`：Agent 执行后遍历最后一次工具调用，解析出结构化提问后追加到回复末尾。

### 亮点

- **从纯文本到结构化**：之前 SSH 失败返回一大段文字说明，现在格式化为带 header + 选项的表单样式
- **识别 10 种提问模式**：覆盖"请向用户"、"请让用户"、"询问用户"、"你可以帮用户安装"等常见引导语
- **CLI 友好展示**：输出格式带分隔线、编号、选项，用户一眼看懂需要提供什么信息
- **可扩展**：新增场景只需在 `_extract_questions()` 中添加识别规则

## 8. TodoWrite 任务追踪

### 如何实现

在 `state.py` 中新增 `TodoList` + `todo_write_tool()`：

```python
class TodoList:
    items: List[TodoItem]  # subject + status (pending/in_progress/completed)
    def set_todos(todos)   # 替换整份清单
    def update(index, status)  # 更新单项
    def format() -> str    # 格式化展示

def todo_write_tool() -> StructuredTool  # LLM 可调用的工具
```

将 `todo_write` 注册到 `ALL_TOOLS`，LLM 在执行复杂任务时可自主调用：

```
用户: "帮我全面排查服务器问题"
LLM → todo_write([
  {"subject": "检查服务器整体资源", "status": "pending"},
  {"subject": "获取 Top 进程", "status": "pending"},
  {"subject": "查看系统日志", "status": "pending"},
])
```

### 亮点

- **任务可视化**：展示 `✓ 已完成 / ◉ 进行中 / ○ 等待中`，用户能实时看到进度
- **LLM 自主规划**：不再依赖模板匹配，LLM 可以根据任务复杂度自主拆分步骤
- **与 ExecutionPlan 互补**：模板匹配用 `ExecutionPlan`（有工具名/参数/结果），自主规划用 `TodoList`（轻量灵活）
- **会话内轻量**：不持久化，专注于当前任务的进度追踪
